## Résumé d’alignement avec enjoyApi (voir fiche backend pour le détail)

> Référence backend : [`../../../enjoyApi/AI_MEMORY.md`](../../../enjoyApi/AI_MEMORY.md). Doc REST côté API : à lier depuis le dépôt enjoyApi (ex. `documentation-api-rest.md`) sans tout dupliquer ici.

- **Stack backend (rappel)** : Java 21, Spring Boot 3.5.6, API sous `/api/v1`, erreurs centralisées (`GlobalExceptionHandler`, etc.) — détail dans `enjoyApi/AI_MEMORY.md`.
- **Activités** (`/sejours/{sejourId}/activites`) : `CreateActiviteRequest` / `UpdateActiviteRequest` exigent **`membreTokenIds`** et **`groupeIds` non vides**, et **`typeActiviteId`** (type du **même séjour**) ; chaque `groupeId` doit être un groupe **du même séjour**. **`enfantIds`** optionnel : IDs SQL des enfants **inscrits au séjour** ; liste **complète** en PUT (`[]` = aucun participant) ; réponse **`ActiviteDto.enfants`** (**`EnfantParticipantInfo`** : id, nom, prénom). **Conflit enfant** (même jour, moments qui se chevauchent dans la hiérarchie) : validation côté front **`idsEnfantsDejaAffectesAutreActivite`** avant enregistrement modale participants. **Date** : doit être dans la période **calendaire** `dateDebut`–`dateFin` du séjour (inclus) ; sinon le backend renvoie une erreur métier (ex. `IllegalArgumentException` → 400). **Animateurs** : le **directeur du séjour** peut figurer dans `membreTokenIds` même sans entrée dans l’équipe (`SejourEquipe`) ; les **autres** tokens doivent correspondre à des membres de l’équipe du séjour. **Conflit animateur** (même jour, moments qui se chevauchent dans la hiérarchie) : **`ANIMATEUR_DEJA_AFFECTE_CRENEAU`** (400) ; front : **`premierConflitAnimateurActiviteInterne`** (`listeActivitesUtils.ts`, **`idsEnConflit`**) avant submit modale. **`lieuId`** optionnel : occupation du lieu **par jour et moment** (chevauchement hiérarchique des moments) selon règles backend (lieu non partageable = une activité ; partageable = plafond `nombreMaxActivitesSimultanees`) ; front : **`verifierDisponibiliteLieuActivite`** (aperçu live + blocage submit, messages **« Occupé par : … »**) ; **`avertissementLieu`** peut être renvoyé sur la réponse POST/PUT (lieu déjà utilisé le même jour mais partage autorisé). Voir `enjoyApi/AI_MEMORY.md` (occupation lieu, `ActiviteDto`, **`TypeActivite`**).
- **Dossiers sanitaires (agrégat)** : **`GET /sejours/{sejourId}/dossiers-enfants`** — agrégation des infos dossier enfants pour l’écran séjour **Sanitaire** ; types **`GroupeResumeDto`**, **`EnfantDossierSanitaireLigneDto`** (`api.d.ts`) ; appel front **`listerDossiersEnfantsSanitaire`** (`sejour-enfant.service.ts`). **UI dossiers** : filtres barre **`DetailsSejourSanitaire`** (7 groupes + **Tout**, logique **OU**, prefs **`enjoy.sanitaire.opts.{sejourId}`**) → **`ListeSanitaireDossiers`** (colonnes dynamiques, **`canPrint`**, métadonnées impression : effectif séjour + filtres **`Liste`**).
- **Cahier d’infirmerie** : **`GET/POST /sejours/{sejourId}/cahier-infirmerie`**, **`PUT/DELETE …/cahier-infirmerie/{entreeId}`**, **`GET …/historique`** — types **`CahierInfirmerieEntreeDto`** / **`SaveCahierInfirmerieEntreeRequest`** (`localisationCorps` optionnel, `temperatureCelsius` si soin **PRISE_TEMPERATURE**) ; côté JSON, **`dateHeure`** peut être **ISO ou nombre** (epoch) → affichage liste via **`formaterDateHeureListeCahier`** (date + heure sur 2 lignes) et **`parseDate`**. Front : **`cahier-infirmerie.service.ts`**, **`ListeCahierInfirmerie`** (colonnes **Localisation**, **Soigné(e) par** ; recherche sans accents ; impression filtrée via **`src/print/`**, colonne **Signature de la direction**), **`CahierInfirmerieForm`** ; historique **`HistoriqueModificationListeModal`** avec **`formatSnapshots="cahier_infi"`** (voir **`libelleHistoriqueModification.ts`**).
- **Lieux** (`/sejours/{sejourId}/lieux`) : **`SaveLieuRequest`** / **`LieuDto`** incluent **`partageableEntreAnimateurs`**, **`nombreMaxActivitesSimultanees`** (obligatoire et ≥ 2 si partage ; sinon `null`), et **`usages: UsageLieu[]`** (**obligatoire**, non vide — `ACTIVITE`, `SURVEILLANCE`, `RASSEMBLEMENT`). Validation côté API : 400 si partage incohérent ou **aucun usage valide** après filtrage (`IllegalArgumentException`). **Règle backend** : source planning **`LIEU`** (libellé ligne ou contenu cellule) impose que chaque lieu référencé ait au moins l'usage **`SURVEILLANCE`** | **`RASSEMBLEMENT`** ; sinon **400** (`IllegalArgumentException`). UI : **`ListeActivites`** filtre automatiquement les lieux avec usage **`ACTIVITE`** ; **`ListePlanningsOrganisation`** filtre les lieux avec usage **`SURVEILLANCE`** | **`RASSEMBLEMENT`**.
- **Équipe** : lors du `DELETE /sejours/{id}/equipe/{membreTokenId}`, le backend **retire automatiquement** ce membre des **référents** de tous les groupes du séjour (à garder en tête pour l’UI : pas seulement « hors équipe », aussi « plus référent »). **Privilèges** : endpoint réservé à `ROLE_DIRECTION` ; un `BASIC_USER` (animateur, AS, etc.) reçoit **HTTP 403** avec un body JSON où le **message métier** est souvent dans **`message`** et une synthèse dans **`error`** — l’UI doit préférer **`message`** lorsqu’il est renseigné (**`getApiErrorMessage`** / **`adaptAxiosError`** dans `helpers/axiosError.ts`). L’UI doit **toujours** afficher l’erreur (voir [decisions-architecturales.md](decisions-architecturales.md), section *Suppression — règle de remontée des erreurs*).
- **Types TS** : `ActiviteDto` (**`moment`**, **`typeActivite`**, **`lieu`**, **`avertissementLieu`**, **`groupeIds`**, **`enfants?: EnfantParticipantInfo[]`**), `CreateActiviteRequest` / `UpdateActiviteRequest` (**`lieuId`**, **`momentId`**, **`typeActiviteId`** obligatoire ; **`enfantIds?`** ; **`null`** sur `lieuId` en mise à jour retire le lieu), **`EnfantParticipantInfo`**, **`TypeActiviteDto`** / **`SaveTypeActiviteRequest`**, **`MomentDto`** / **`SaveMomentRequest`**, **`HoraireDto`** / **`SaveHoraireRequest`** (libellé format `6h00`–`23h59`), **`UsageLieu`** (`ACTIVITE` | `SURVEILLANCE` | `RASSEMBLEMENT` — enum `enums/UsageLieu.ts`), `LieuDto` / `SaveLieuRequest` (**partage**, **`usages: UsageLieu[]`** obligatoire) dans `api.d.ts` — section **Types d’activité** (par séjour) documentée avant **Activités** lorsque pertinent. **`PlanningCelluleDto`** / **`PlanningCellulePayload`** : références cellule en **tableaux** (`horaireIds`, `horaireLibelles`, `momentIds`, `groupeIds`, `lieuIds`, `membreTokenIds`) — voir **`frontend-planning-cellules-multiples.md`** enjoyApi.
- **Horaires** (`/sejours/{sejourId}/horaires`) : libellés d’heure par séjour (ex. `8h30`). **`SaveHoraireRequest.libelle`** : pattern aligné **`Horaire.LIBELLE_HORAIRE_PATTERN`** côté API ; unicité par séjour **insensible à la casse** → **409** si doublon. Pas de lien avec **`Activite`** dans l’API à ce stade (suppression sans garde-fou métier). Voir `enjoyApi/AI_MEMORY.md` (endpoints horaires).
- **Plannings direction (grilles)** (`/sejours/{sejourId}/planning-grilles`) : **`PlanningGrille`**, **`PlanningLigne`**, **`PlanningCellule`** — détail des entités, `sourceLibelleLignes` / `sourceContenuCellules`, **cellules** avec **plusieurs ids** par type (payload / DTO **`horaireIds`**, **`horaireLibelles`**, **`momentIds`**, **`groupeIds`**, **`lieuIds`**, **`membreTokenIds`**) ; doc front **sync** : **`enjoyApi/docs/frontend-planning-cellules-multiples.md`**. Tests côté API dans `enjoyApi/AI_MEMORY.md` ; doc backend **`enjoyApi/docs/planning-grilles-api.md`**.
- **Activités prestataires (sorties / prestas)** (`/sejours/{sejourId}/activites-prestataires`) : **`SaveActivitePrestataireRequest`** — **`nom`**, **`date`** (période séjour), **`momentIds`** (≥ 1), **`groupeIds`** optionnel, **`nonParticipations`** (`tokenId` + `momentId`). **Anti-doublon** (POST/PUT) : interdit deux sorties distinctes pour le même triplet **date × moment × groupe** ; **`groupeIds` vide** → pas de contrôle doublon (sortie visible surtout sur la fiche liste). **400** : message métier du type *« Une sortie est déjà planifiée pour le groupe « … » le … au moment « … ». »* — afficher tel quel (**`adaptAxiosError`** + **`preserveResponseData`** sur POST/PUT). **Conflit sortie ↔ activité interne** (même référent / date / moment) : géré à part via **`nonParticipations`** et éventuelle suppression d’activité interne (dialogue direction). **Historique** : **`GET …/activites-prestataires/{activitePrestataireId}/historique`** — **`HistoriqueModificationActivitePrestataireDto`** (**`type: ACTIVITE_PRESTATAIRE`**, snapshots `Nom: … | Date: … | …`, **`@JsonUnwrapped`** à plat) ; traçabilité automatique côté API sur POST/PUT/DELETE (aucun payload historique au front). Front : **`getHistoriqueActivitePrestataire`**, modal **`formatSnapshots="activite_prestataire"`** ; non-participations historique : phrases par animateur avec **moments** entre parenthèses. Front : pré-validation **`messageDoublonSortiePrestataire`**, regroupement exclusions **`grouperNonParticipationsParAnimateur`** / tri moments **`trierMomentsChronologiquement`** (`listeActivitesPrestatairesCalendrier.ts`).
- **Chambres (hébergement)** (`/sejours/{sejourId}/chambres`) : ressource distincte des **Lieux**. **`ChambreDto`** : **`typeChambre`**, **`identifiant`**, **`nom`**, **`capaciteMax`**, **`genreAutorise`**, localisation (**`batiment`**, **`couloir`**, **`etage`**), **`groupe`** (**`GroupeResumeDto`**, type **ENFANT** ; `null` = sans restriction), **`referents`**, **`occupants`**. **Lecture et écriture** (CRUD, référents, occupants) : privilège **`ACCES_SEJOUR`** — tout membre rattaché au séjour (directeur, ANIM, AS, SB, AUTRE, ADJOINT) ; hors séjour → **403**. **Historique** : **`GET …/chambres/{chambreId}/historique`** — **`HistoriqueModificationChambreDto`** (snapshots lisibles `Type: … | Identifiant: … | …`, **`@JsonUnwrapped`** à plat) ; alimenté côté API à chaque mutation chambre / référent / occupant. Front : **`getHistoriqueChambre`**, modal partagée **`formatSnapshots="chambre"`**, diff occupants (ajouts/retraits, accord genre). **Référents** : **`POST/DELETE …/referents`** — **ENFANT** only. **Occupants** : **`POST/DELETE …/occupants/enfants`** (unitaire ou batch) ; **`POST/DELETE …/occupants/equipe`**. Règles : un occupant = une chambre max (réaffectation = déplacement) ; genre / capacité / **`groupeId`** / **`numeroLit`** (409 doublon lit). **PUT** : changement **`typeChambre`** efface occupants ; **ENFANT → EQUIPE** efface référents + groupe. Erreurs : **400** métier, **404**, **409** identifiant ou lit. Tri liste : **`batiment`**, **`etage`**, **`couloir`**, **`identifiant`** ; occupants triés **`numeroLit`**, nulls en fin.

## Alignement types front

- Types partagés : `src/types/api.d.ts` — rester aligné avec les DTO / payloads documentés côté API.
- Références utilisateur côté JSON : **`tokenId`**, pas l’id SQL interne (conventions enjoyApi).
- **Utilisateurs** (`PUT /utilisateurs`, body **`UpdateUserRequest`**) : modification du champ **`email`** soumise aux droits backend — **ADMIN** (toute cible, y compris soi) ; **DIRECTION** sur un autre **`BASIC_USER`** (ex. membre d’équipe via **`UserForm`** + **`sejourId`**) ; interdit en auto-modification **DIRECTION** / **BASIC_USER**. Le front **envoie toujours `email`** dans le payload (valeur existante si champ UI désactivé). Erreurs typiques : **403**, **409**, **400**, **404** — affichage via **`getApiErrorMessage`** (**`Profil.tsx`**, **`UserForm.tsx`**). Helper UI : **`canEditEmail`** (`helpers/canEditEmail.ts`).
- **Photo de profil** (`/utilisateurs/{tokenId}/photo-profil`) : **`POST`** multipart (**`file`**, JPEG/PNG/WebP, max **2 Mo**) → **`ProfilUtilisateurDTO`** ; **`GET`** → binaire (affichage front via blob URL Axios, **pas** **`photoProfilUrl`** direct en **`<img>`**) ; **`DELETE`** → **204**. **`ProfilUtilisateurDTO.photoProfilUrl`** : indicateur de présence (URL serveur, non utilisée pour le rendu). Front : **`remplacerPhotoProfil`** = DELETE puis POST (**404** sur DELETE ignoré) ; cadrage « brûlé » dans le JPEG exporté (pas d’offset API).

## Services API (inventaire)

- **caller.service.ts** : Instance Axios configurée avec intercepteurs
  - Base URL : `import.meta.env.VITE_API_URL` ou `http://localhost:8080/api/v1` par défaut
  - Intercepteur request : Ajoute automatiquement le token Bearer si l'utilisateur est connecté
  - Intercepteur response : **401** → tentative refresh ; échec ou session invalide → mémorisation URL (**`cheminApresConnexion`**), logout, **`window.location.replace('/')`** (sauf échec login sur **`/`** avec **`X-Skip-Token-Refresh`**)
  - `withCredentials: true` activé par défaut pour les cookies
- **account.service.ts** : Gestion de l'authentification
  - `login()` : Connexion avec header `X-Skip-Token-Refresh`
  - `addUser()` : Inscription avec header `X-Skip-Token-Refresh`
  - `refreshAccessToken()` : Rafraîchissement du token d'accès (**`POST /auth/refresh-token`** avec **`X-Skip-Token-Refresh`**)
  - `getTokenInfo()` : Décodage du JWT pour récupérer les infos utilisateur
  - `saveAccessToken()` : Sauvegarde du token chiffré dans localStorage avec CryptoJS
  - `getToken()` : Récupération et déchiffrement du token depuis localStorage
  - `logout()` : Suppression du token et nettoyage du store Redux
  - `isLogged()` : Vérification de la présence d'un token
- **sejour.service.ts** : CRUD des séjours uniquement
  - `getAllSejours()`, `getSejourById()`, **`getAllSejoursByUtilisateur()`** (endpoint **`/sejours/utilisateur/{utilisateurTokenId}`** — accessible ADMIN, DIRECTION, BASIC_USER ; le backend filtre selon le rôle), `addSejour()`, `updateSejour()`, `deleteSejour()`
  - Tri automatique de l'équipe par nom dans `getSejourById()` via **`trierUtilisateursParNom`** (`helpers/trierUtilisateurs.ts`)
  - Utilise `validateResponseStatus` et `adaptAxiosError` de `helpers/axiosError.ts`
- **sejour-equipe.service.ts** : Gestion de l'équipe d'un séjour
  - `ajouterNouveauMembreEquipe()`, `ajouterMembreExistantEquipe()`, `modifierRoleMembreEquipe()`, `supprimerMembreEquipe()`
- **sejour-enfant.service.ts** : Gestion des enfants d'un séjour
  - `getEnfantsDuSejour(sejourId)` : Liste triée par nom via `trierEnfantsParNom`
  - `listerDossiersEnfantsSanitaire(sejourId)` : `GET /sejours/{sejourId}/dossiers-enfants` — agrégat pour la liste **Sanitaire** (réponses **`EnfantDossierSanitaireLigneDto`**)
  - `getDossierEnfant()`, `updateDossierEnfant()` : Dossier enfant (contrôle d'accès 403 si non participant)
  - `creerEtAjouterEnfant()`, `modifierEnfant()`, `supprimerEnfantDuSejour()`, `supprimerTousLesEnfants()`
  - `getExcelImportSpec(sejourId)` : Récupère la notice d'import Excel (colonnes obligatoires, optionnelles, mots-clés, formats) — `GET /sejours/{sejourId}/enfants/import/spec`
  - `importerEnfantsExcel()` : Import Excel (FormData, retourne `ExcelImportResponse`)
  - Gestion erreurs validation (400) avec `preserveResponseData: true` pour les formulaires
- **cahier-infirmerie.service.ts** : Cahier d’infirmerie du séjour — `listerEntrees(sejourId)`, `creerEntree`, `modifierEntree`, `supprimerEntree` (204 attendu), `getHistorique(sejourId, entreeId)` ; préfixe **`/sejours/{sejourId}/cahier-infirmerie`**
- **sejour-groupe.service.ts** : Gestion des groupes d'un séjour (export `sejourGroupeService`)
  - `getGroupesDuSejour(sejourId)`, `getGroupeById(sejourId, groupeId)`
  - `creerGroupe(sejourId, request)` : Crée un groupe (POST). Pour AGE/NIVEAU_SCOLAIRE, le backend ajoute automatiquement les enfants. Fallback frontend si groupe créé vide.
  - `modifierGroupe()`, `supprimerGroupe()`, `ajouterEnfantAuGroupe()`, `retirerEnfantDuGroupe()`
  - `ajouterReferent(sejourId, groupeId, request)` : `POST /sejours/{sejourId}/groupes/{groupeId}/referents` — body `AjouterReferentRequest` (`referentTokenId`)
  - `retirerReferent(sejourId, groupeId, referentTokenId)` : `DELETE /sejours/{sejourId}/groupes/{groupeId}/referents/{referentTokenId}`
- **sejour-activite.service.ts** : Activités d’un séjour (export `sejourActiviteService`)
  - `getActivitesDuSejour(sejourId)` : `GET /sejours/{sejourId}/activites` (réponses avec **`moment`**, **`lieu`** ; **`avertissementLieu`** souvent `null` hors POST/PUT)
  - `getActiviteById(sejourId, activiteId)` : `GET /sejours/{sejourId}/activites/{activiteId}`
  - `creerActivite(sejourId, request)` : POST, attend 201, body peut inclure **`lieuId`**, **`momentId`** ; réponse peut contenir **`avertissementLieu`**
  - `modifierActivite()`, `supprimerActivite()` : PUT (peut envoyer **`lieuId: null`**, **`momentId`**) / DELETE (204 attendu pour suppression)
- **sejour-activite-prestataire.service.ts** : Sorties / prestataires (export **`sejourActivitePrestataireService`**) — préfixe **`/sejours/{sejourId}/activites-prestataires`**
  - `listerActivitesPrestataires(sejourId)` : GET — chargé par **`detailsSejourLoader`**
  - `getActivitePrestataire`, `creerActivitePrestataire`, `modifierActivitePrestataire`, `supprimerActivitePrestataire` : CRUD ; POST/PUT avec **`preserveResponseData: true`** (messages métier 400, ex. doublon date/moment/groupe)
  - `getHistoriqueActivitePrestataire(sejourId, activitePrestataireId)` : GET **`…/activites-prestataires/{id}/historique`** → **`HistoriqueModificationActivitePrestataireDto[]`**
- **sejour-moment.service.ts** : Moments d’un séjour (export `sejourMomentService`)
  - `getMomentsDuSejour(sejourId)` : `GET /sejours/{sejourId}/moments`
  - `getMomentById(sejourId, momentId)` : `GET /sejours/{sejourId}/moments/{momentId}`
  - `creerMoment(sejourId, request)` : POST, attend 201, body `SaveMomentRequest` (`nom`, `parentId?`)
  - `modifierMoment(sejourId, momentId, request)` : PUT
  - `supprimerMoment(sejourId, momentId)` : DELETE, attend 204
  - `reordonnerMoments(sejourId, request)` : `PUT /sejours/{sejourId}/moments/reorder`, attend 200, body `ReorderMomentsRequest`, retourne `MomentDto[]`
- **sejour-lieu.service.ts** : Lieux d’un séjour (export `sejourLieuService`)
  - `getLieuxDuSejour(sejourId)`, `getLieuById(sejourId, lieuId)` : GET (erreurs propagées / console) — **`LieuDto.usages`** renvoyé
  - `creerLieu(sejourId, request)` : POST, attend 201 (`SaveLieuRequest` incl. **partage**, **`usages`** obligatoire)
  - `modifierLieu(sejourId, lieuId, request)` : PUT (même body que création, incl. **`usages`**)
  - `supprimerLieu(sejourId, lieuId)` : DELETE, attend 204
- **sejour-chambre.service.ts** : Chambres d’hébergement d’un séjour (export **`sejourChambreService`**) — préfixe **`/sejours/{sejourId}/chambres`**
  - `getChambresDuSejour(sejourId)`, `getChambreById(sejourId, chambreId)` : GET
  - `creerChambre(sejourId, request)` : POST, attend 201 (`SaveChambreRequest`)
  - `modifierChambre(sejourId, chambreId, request)` : PUT
  - `supprimerChambre(sejourId, chambreId)` : DELETE, attend 204
  - `ajouterReferent(sejourId, chambreId, request)` : POST **`…/referents`**, body **`AjouterReferentRequest`**
  - `retirerReferent(sejourId, chambreId, referentTokenId)` : DELETE **`…/referents/{referentTokenId}`**, attend 204
  - `affecterEnfant`, `affecterEnfants`, `retirerEnfant` : POST/DELETE **`…/occupants/enfants`** (unitaire ou lot)
  - `affecterMembreEquipe`, `affecterMembresEquipe`, `retirerMembreEquipe` : POST/DELETE **`…/occupants/equipe`**
  - `getHistoriqueChambre(sejourId, chambreId)` : GET **`…/chambres/{chambreId}/historique`** → **`HistoriqueModificationChambreDto[]`**
- **sejour-horaire.service.ts** : Horaires d’un séjour (export `sejourHoraireService`)
  - `getHorairesDuSejour(sejourId)` : `GET /sejours/{sejourId}/horaires` (ordre id côté API ; l’UI applique **`trierHorairesChronologiquement`**)
  - `getHoraireById(sejourId, horaireId)` : `GET .../horaires/{horaireId}`
  - `creerHoraire(sejourId, request)` : POST, attend 201, body `SaveHoraireRequest`
  - `modifierHoraire(sejourId, horaireId, request)` : PUT (409 si libellé déjà pris par un autre horaire du séjour)
  - `supprimerHoraire(sejourId, horaireId)` : DELETE, attend 204
- **sejour-reunion.service.ts** : Comptes rendus de réunion (export `sejourReunionService`) — préfixe **`/sejours/{sejourId}/reunions`**
  - `listerReunions(sejourId)` : GET — liste **`ReunionDto[]`** (chargée aussi par **`detailsSejourLoader`**)

- **utilisateur.service.ts** : Gestion des utilisateurs
  - Tri automatique par nom dans plusieurs méthodes (`getAllUsers()`, `getDirecteurs()`, `getEquipeBySejour()`)

  - `uploadPhotoProfil(tokenId, file)` : **`POST /utilisateurs/{tokenId}/photo-profil`** (multipart, validation type/taille client)
  - `getPhotoProfilBlobUrl(tokenId, cacheBust?)` : **`GET`** en **`blob`** → **`URL.createObjectURL`** (404 → `null`)
  - `deletePhotoProfil(tokenId)` : **`DELETE`**, attend **204**
  - `remplacerPhotoProfil(tokenId, file)` : DELETE (404 ignoré) puis **`uploadPhotoProfil`**
  - Utilise `adaptAxiosError` et `validateResponseStatus` pour `deleteUser()` et `changePassword()`

## Glossaire

- **Directeur :** Rôle utilisateur (`RoleSysteme.DIRECTION`) gérant des séjours spécifiques. Accès aux pages **`/mes-sejours/*`** (avec les BASIC_USER membres d'équipe). Voit les séjours dont il est directeur **ou** dont il est membre d'équipe.
- **Animateur (BASIC_USER) :** Rôle utilisateur (`RoleSysteme.BASIC_USER`) — membre d'équipe d'un ou plusieurs séjours. Accès aux pages **`/mes-sejours/*`** ; ne voit que les séjours dont il est membre d'équipe (filtrage côté backend).
- **Admin :** Super-utilisateur (`RoleSysteme.ADMIN`) gérant l'ensemble de la plateforme. Accès aux pages `/utilisateurs`, `/sejours`.
- **Séjour :** Entité principale du système contenant nom, description, lieu, dates, équipe, enfants, plannings.
- **Sortie / prestataire :** Activité externalisée (`ActivitePrestataireDto`) — date, moments, groupes optionnels, **`nonParticipations`** (animateur exclu du calendrier pour un créneau). Visible sur le planning des référents des groupes concernés et sur **`/activites/sorties-prestas`**. UI exclusions : **un nom d’animateur** + moments (cases ou liste) ; historique **`ACTIVITE_PRESTATAIRE`** via **`HistoriqueModificationListeModal`**.
- **Équipe :** Liste des utilisateurs assignés à un séjour (composant `Equipe.tsx`).
- **Enfant :** Entité représentant un enfant avec ses informations personnelles (nom, prénom, genre, date de naissance, niveau scolaire). Un enfant peut être réutilisé dans plusieurs séjours. API genre : **`Masculin`** / **`Féminin`** ; UI liste **`ListeEnfants`** : affichage **Garçon** / **Fille** (**`libelleGenreEnfant`**). Composants : `ListeEnfants.tsx` (colonnes masquables via **`Liste`** **`toggleable`**, impression filtrée **`canPrint`** + sélection personnalisée **`canPrintCustomSelection`**), `AddEnfantForm.tsx`, `ImportExcelEnfants.tsx`.
- **Impression (module front) :** Dossier **`src/print/`** — impression ciblée via **`react-to-print`** (`usePrintContent`, **`PrintContentRoot`**, **`PrintTrigger`**). En-tête répété par page **Titre — n/N** (**13pt**, gras) par défaut ; **`runningHeaderLayout: 'title-page-split'`** (opt-in **`DossierEnfant`**) : titre à gauche, **n/N** à droite. **`PrintDocumentHeader`** = métadonnées utiles seulement (pas de redondance avec choix écran : période, vue, rendu couleur/N&B, colonnes cochées). **Pagination multi-pages** : **`buildPrintPageStyle`** force **`html`/`body`** en **`height: auto`**, **`overflow: visible`**. Listes séjour : **`ListeEnfants`**, **`TableauUtilisateurs`**, **`ListeSanitaireDossiers`** ; **cahier d'infirmerie** : **`ListeCahierInfirmerie`** ; **chambres** : **`ListeChambres`** ; **fiche dossier** : **`DossierEnfant`** ; **planning activités (calendrier)** : **`ListeActivites`** (modale **Noir et blanc** / **Couleurs**, preset **`activitesCalendarGrid`**) ; **menus** : **`DetailsSejourMenus`** (modale **Noir et blanc** / **Couleurs**, **`couleurFondCarteMenuPourTypeRepas`**, preset **`menusGrid`**) ; **plannings organisation** : **`ListePlanningsOrganisation`** — **`PlanningGrillePrintTable`**, impression **unitaire** + **multiple** (preset **`planningGrid`**, sans consignes à l’impression). Voir [decisions-architecturales.md](decisions-architecturales.md) (section *Module impression*).
- **Profil utilisateur :** Page **`/profil`** — **`Profil.tsx`** : édition personnelle champ par champ ; carte rectangulaire (photo via blob API + **badge rôle** **`libelleRoleBadgeProfil`**) ; upload/recadrage (**`PhotoProfilRecadrageModal`**, **`photoProfilRecadrage.ts`**) ; lightbox ; accordéons Informations / Contact / Compte ; restrictions email (**`canEditEmail`**). **Header** : même libellé de rôle à côté du prénom.
- **Contexte séjour header :** **`helpers/headerSejourContext.ts`** — snapshot **`enjoy.headerSejourContext`** (session) + **`enjoy.dernierSejourId.{sub}`** (local) ; alimente la nav séjour et le **rôle affiché** Profil / Header.
- **Groupe :** Regroupement d'enfants d'un séjour. Types : THEMATIQUE (manuel), AGE (tranche d'âge), NIVEAU_SCOLAIRE (tranche de niveau). Composants : `ListeGroupes.tsx`, `CreateGroupeForm.tsx`, affichage pastilles **`AffichageGroupesListe`** ; **sélection** (modales, filtres, `<select>`) via **`SelectionGroupesParType`** + **`groupesParType.ts`** (sections par **`typeGroupe`**, tri **nom** alpha). Service : `sejour-groupe.service.ts` (export `sejourGroupeService`). Pour AGE/NIVEAU_SCOLAIRE, ajout automatique à la création (backend) ou fallback frontend ; bouton « Ajouter les enfants de la tranche » pour ajouter en masse les enfants correspondants.
- **Référent (groupe) :** Membre de l'équipe du séjour associé à un groupe comme point de contact. Géré via `ReferentsSelector` + appels `ajouterReferent` / `retirerReferent` ; listé dans `GroupeDto.referents` (`ReferentInfos`).
- **Chambre (séjour) :** Hébergement rattaché au séjour (`ChambreDto`), distinct du **Lieu**. Types **`ENFANT`** / **`EQUIPE`**. **Droits** : **`ACCES_SEJOUR`** pour lecture et écriture (tous les membres du séjour). UI : **`ListeChambres.tsx`** (impression liste filtrée via **`src/print/`**, libellés genre print **`GenreChambrePrintLabels`**) ; helpers **`chambreOccupantsUtils.ts`** (**`fusionnerChambreRetourneeDansListe`** après POST affectation), **`libelleChambre.ts`** (**`textesRechercheChambre`**, **`chambreCorrespondRechercheTexte`**) ; historique **`HistoriqueModificationListeModal`** (**`formatSnapshots="chambre"`**).
- **Dossier enfant :** Informations complémentaires d'un enfant (contacts parents, infos médicales, PAI, alimentaires, traitements). Page **`DossierEnfant`** — route **`/mes-sejours/:sejourId/enfants/:enfantId/dossier`** (icône dossier **`ListeEnfants`**, **`ListeSanitaireDossiers`**) ; édition par section (**`peutModifierDossierEnfant`**) ; **Imprimer** fiche complète (**`src/print/`**). Accès : participants du séjour.
- **Référence alimentaire :** Entrée du référentiel global (allergène ou régime/préférence). `ReferenceAlimentaireDto` (`type`, `libelle`, **`ordre`**, `actif`) ; CRUD **`references-alimentaires.service.ts`**. UI catalogue : **`ListeReferencesAlimentaires`** (Paramétrage accordéon **5**) — **`ordre`** sert au tri (`trierReferencesAlimentaires`) mais n’est pas saisi ni affiché (assignation auto à la création). Liens dossier enfant, formulaires menus, **`GET …/references-alimentaires-agregees-enfants`**.
- **Loader :** Fonction asynchrone React Router pour charger les données avant le rendu. Erreurs bloquantes (réseau, auth, serveur, 404…) : **`throwRouteLoaderError`** (`helpers/routeError.ts`) → boundary **`error-page`** ou page **`/erreur`**. Exemple **`detailsSejourLoader`** (route **`sejour-detail`**) : **`Promise.all`** séjour, enfants, groupes, **chambres**, lieux, moments, horaires, activités, types d’activité, **`reunions`**, **`planningGrilles`**, **`activitesPrestataires`** — enregistre le séjour dans **`sejourDetailRouteCache`** pour les loaders enfants. **`chambres`** alimente la vue générale (**`ListeEnfants`**, **`Equipe`**) en plus de l’onglet **`chambresLoader`**. **`menusLoader`** : 2 GET (références alimentaires agrégées + menus sur la plage du séjour), sans 2ᵉ fetch séjour. **`sanitaireLoader`** : dossiers + cahier infirmerie. **`chambresLoader`** : liste **`GET …/chambres`** (onglet Chambres).
- **Moment (séjour) :** Créneau nommé rattaché au séjour (ex. Matin, Après-midi). `MomentDto` (**`ordre`** pour l’ordre d’affichage, **`parentId: number | null`** pour la hiérarchie) ; CRUD + **réordonnancement** (`reordonnerMoments`) via `ListeMoments` et `sejourMomentService`. **Hiérarchie** : un moment peut avoir un parent unique et plusieurs enfants (profondeur libre) ; helper `construireArbreMoments.ts` — **`idsEnConflit`** (paramètre **`readonly MomentDto[]`**), `aplatirMomentsHierarchiquement`, `validerLiensParentEnfants`, etc. **`trierMomentsChronologiquement`** : si `parentId` présent → parcours hiérarchique ; sinon tri `ordre / id`. **UI** : accordéons sur les sous-moments (bouton ▼/►, `accordionsFermés`), sélecteurs indentés dans les formulaires activités / plannings. Les **activités** référencent un moment (`ActiviteDto.moment`, `momentId`) ; règle backend : impossible d’affecter un animateur à deux activités le même jour sur des moments qui se chevauchent dans la hiérarchie (**`ANIMATEUR_DEJA_AFFECTE_CRENEAU`** 400) ; pré-validation modale **`ListeActivites`**. Sans au moins un moment, l’UI bloque la création d’activités.
- **Horaire (séjour) :** Libellé d’heure rattaché au séjour (ex. `8h30`, format `6h00`–`23h59`). `HoraireDto` ; CRUD via `ListeHoraires` et `sejourHoraireService`. Affichage trié chronologiquement (`trierHorairesChronologiquement`). Non référencé par les **activités** dans l’API à ce stade.
- **Lieu (séjour) :** Espace ou salle rattaché au séjour (`LieuDto` : nom, `EmplacementLieu`, capacité max optionnelle **`nombreMax`**, **partage** : **`partageableEntreAnimateurs`**, **`nombreMaxActivitesSimultanees`** si partage ≥ 2). UI : `ListeLieux.tsx` ; API : `sejour-lieu.service.ts`.
- **Activité (séjour) :** Action planifiée (date dans l’intervalle du séjour, nom, description, **créneau** `moment` / `momentId`, **au moins un groupe** `groupeIds`, **au moins un animateur** `membreTokenIds`, **lieu optionnel** `lieuId` / `ActiviteDto.lieu`, **enfants participants optionnels** `enfantIds` / **`ActiviteDto.enfants`**). Le directeur peut être animateur hors équipe ; les autres doivent être dans l’équipe. Conflits **animateur**, **enfant** (même créneau hiérarchique) et **occupation lieu** : pré-validation front + contrôle API. UI : **`ActiviteEnfantsParticipantsModal`** (icône liste/calendrier, hors modale CRUD) ; module **`ListeActivites*.tsx`** sur **`DetailsSejourActivites`** ; API : `sejour-activite.service.ts`.
- **Planning organisation (grille) :** Grille de planning **direction** pour un séjour (lignes × jours du séjour). **Impression** : période et vue (**1 / 3 / 7 jours**) = fenêtre affichée (**`joursFenetre`**), pas l’intégralité du séjour ; **unitaire** (éditeur) ou **multiple** (liste **`/organisation`**, ordre personnalisable) via **`PlanningGrillePrintTable`** — **sans** consignes ni métadonnées en-tête à l’impression. **Cellules** : selon `sourceContenuCellules`, contenu = **listes** d’ids (**`horaireIds`**, **`momentIds`**, **`groupeIds`**, **`lieuIds`**) et/ou **`membreTokenIds`** / **`texteLibre`** (`SAISIE_LIBRE`) — **`PlanningCelluleDto`** expose aussi **`horaireLibelles`** (même ordre que les ids horaires côté API). **Libellé de ligne** peut être un **membre d’équipe** via `sourceLibelleLignes === "MEMBRE_EQUIPE"` et **`libelleUtilisateurTokenId`** (champs singuliers **réservés aux lignes**, pas aux cellules). API : **`sejour-planning-grille.service.ts`** ; types **`PlanningGrilleDetailDto`**, **`PlanningGrilleSummaryDto`**, **`PlanningLigneDto`**, **`PlanningCelluleDto`**, **`PlanningCellulePayload`**, etc. dans **`api.d.ts`**. UI : **`ListePlanningsOrganisation`** (panneau accordéon **`10`**, props loader **`grilles`**, **`enumererJoursSejour`**, équipe + directeur pour cellules et libellés **membre**) — affichage cellule **plusieurs lignes** dans la case pour plusieurs valeurs ; **création** grille : **consigne globale** sur **une ligne**. À la définition **`sourceLibelleLignes`** / **`sourceContenuCellules`**, options **sans référents** métier : **disabled** + **`noteCreerDabordPourTypePlanning`** (sauf valeur déjà enregistrée). Distinct du **calendrier des activités** (`ListeActivitesCalendrier`).
- **Action :** Fonction React Router pour gérer les soumissions de formulaires (ex: `loginAction`).
